# ドキュメント管理規約

## バイリンガル必須ルール（最重要）

このプロジェクトのすべての公開・共有ドキュメントは **日本語・英語の両方を必須** とする。片方の言語だけで完結するドキュメントは出荷しない。

### 言語の主従関係

- **英語が正（canonical / source of truth）**、日本語は従（翻訳）。
- 内容の追加・変更はまず英語版に入れ、その内容を日本語版へ反映する。

### 同期ルール（絶対）

> **英語ドキュメントを更新したら、対応する日本語ドキュメントを必ず同じ変更（同一コミット）内で更新すること。**

- 英語版だけを更新して日本語版を後回しにしてはならない。「あとで翻訳する」は不可。
- 構造変更（見出し追加・削除、節の並べ替え）も両言語で一致させる。
- 翻訳が即座に難しい場合でも、最低限プレースホルダや要約を入れて両版の構造を一致させ、TODO を残す。
- 逆方向（日本語だけ先に更新）も避ける。正である英語版を先に更新する。

### 言語ペアの対応表

| 英語（正） | 日本語（従） | 同期方法 |
|-----------|-------------|----------|
| `README.md` | `README.ja.md` | 手動で両ファイルを編集 |
| `sphinx-docs/`（英語ソース `.md`） | `sphinx-docs/locale/ja/` の `.po` | 英語ソース変更後に `make update-po` で `.po` を更新し、翻訳を埋める |
| `docs/`（内部設計ドキュメント） | 同上 | 英語で記述し、日本語の要約・対訳を併記する |

### 完了条件（Definition of Done への追加）

- ドキュメント変更を含む作業は、**両言語が揃って初めて完了**とみなす。
- レビュー・コミット前に「英語を変えたら日本語も変えたか？」を必ず確認する。

## ドキュメントの分類

### 1. 永続的ドキュメント（`docs/`）
アプリケーション全体の「何を作るか」「どう作るか」を定義する恒久的なドキュメント。大きな設計変更時のみ更新する。

### 2. 作業単位のドキュメント（`.steering/[YYYYMMDD]-[NNN]-[開発タイトル]/`）
特定の開発作業における「今回何をするか」を定義する一時的なステアリングファイル。作業ごとに新しいディレクトリを作成し、完了後は履歴として保持する。

### 3. ユーザー向けドキュメント（`sphinx-docs/`）

外部公開用のドキュメントサイト。Sphinx + Furo テーマで構築し、GitHub Pages でホスティングする。

- `sphinx-docs/user/` — ユーザーガイド（インストール、クイックスタート、設定、変更履歴）
- `sphinx-docs/dev/` — 開発者ガイド（セットアップ、ビルド、アーキテクチャ、コントリビュート）
- `sphinx-docs/locale/ja/` — 日本語翻訳（`make update-po` で .po ファイルを生成・更新）

**`docs/`（内部設計）と `sphinx-docs/`（外部公開）の役割分担:**
- `docs/` は開発チーム向けの設計・要件ドキュメント（PRD、機能設計書等）
- `sphinx-docs/` はエンドユーザー・コントリビューター向けの公開ドキュメント

## 機能実装時のドキュメント更新

コードの変更を伴う機能実装・バグ修正を行う場合、以下のドキュメントを同時に更新すること:

| ドキュメント | 更新対象 | 必須/任意 |
|-------------|---------|----------|
| `CLAUDE.md` | アーキテクチャセクション | 必須 |
| `README.md`（英語・正） | Features セクション（該当カテゴリ） | 必須 |
| `README.ja.md`（日本語・従） | `README.md` と同じ変更を反映 | 必須（`README.md` を更新したら必ず） |
| `sphinx-docs/` | 該当する機能ページ（英語ソース）＋ `make update-po` で日本語 `.po` | 必須（sphinx-docs が存在する場合、両言語） |
| `docs/product-requirements.md` | 実装済み機能セクション（ロードマップから移動） | 必須（ロードマップ記載機能の場合） |

### 更新タイミング

- ドキュメント更新は **該当機能のコミットに含める**（機能コード + テスト + ドキュメント更新 = 1コミット）
- または、全機能実装後に **1つのドキュメント更新コミット** としてまとめてもよい
- tasklist.md にドキュメント更新タスクを必ず含めること
